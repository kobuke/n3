# Nanjo NFT Wallet — プロダクトの思想からユーザー体験まで

> 対象読者: デザイナー・プランナー・NotebookLM
> 目的: コードベースから読み取れる「思想・コンセプト・UX」を非エンジニアに伝えるための資料
> 作成日: 2026年3月21日
> ※ 推測に基づく部分は `(推測)` と明記しています

---

## 1. プロダクトの概要と思想

### このプロダクトが解決しようとしている課題

**「一度来た旅行者が、二度と戻らない」問題**

一般的な観光地では、訪問者との関係はその日限りで終わる。パンフレットを配っても、スタンプカードを押しても、旅行から帰ると記憶は薄れ、再訪の動機は失われていく。

Nanjo NFT Wallet は、この課題を「デジタルのつながりを持つ」ことで解決しようとしている。旅行者がスマートフォンひとつで受け取れる「デジタル証明書 / チケット / 記念品」を通じて、南城市との関係を訪問後も継続させる仕組みを提供する。

また、これはブロックチェーン技術を活用しているが、**ユーザーにブロックチェーンを意識させないこと**が設計の根幹にある。「メールアドレスだけで始められる」「暗号資産の知識は不要」という方針が、コードの随所に現れている。

### 想定ユーザー像

| ユーザータイプ | どんな人か | どんな状況か |
|---|---|---|
| **観光客（初回）** | 南城市を訪れた旅行者。スマートフォンは普段使い。NFTには馴染みがない。 | 現地でイベント参加、飲食、体験をしながら、記念になるものを受け取る |
| **リピーター・ファン** | 南城市が好きで何度も訪れる人。または移住・二拠点の人。 | コミュニティ（Discord）に参加し、「デジタル住民」としての所属感を楽しんでいる |
| **地元住民** | 南城市在住者。観光客と交流しながら地域活性に関わりたい人。 | 観光客にNFTをギフトしたり、クエスト（スタンプラリー）に参加したりする |
| **スタッフ（運営者）** | 市や観光協会のスタッフ。NFTの発行・管理を行う。技術者ではない可能性もある。 | 配布スポットを設置したり、イベント用のチケットを準備したりする |

### プロダクトのコアとなる価値・思想

**1. 「訪問」を「関係」に変える**
旅行者との関係を「一回きりの消費」で終わらせず、デジタルの証明書・コミュニティ・特典を通じて「継続的なエンゲージメント」へと昇華させる。

**2. 「持っていること」に意味をもたせる**
NFT（デジタルチケット・証明書）は単なるデータではなく、「有効期限」「使用履歴」「ランク」などの概念を付与することで、「保有し続けることに価値がある」状態をつくる。

**3. 「Web3の複雑さ」を包み隠す**
メールアドレスひとつで始められ、お金（暗号資産）のやり取りは基本的に発生しない。ブロックチェーンはバックグラウンドで動いているが、ユーザーは「デジタルチケットを持っている」という感覚で使える。

**4. 既存のインフラ（LINE / Discord / Shopify）に乗る**
ユーザーに新しいアプリのインストールや登録を強いるのではなく、すでに使っているプラットフォームと連携することで摩擦を最小化する。

## 2. ユースケースとストーリー

### ストーリー①：初めて南城市を訪れた観光客がNFTを受け取る

> **ユーザー：** 東京から来た30代の旅行者。スマートフォンは持っているが、NFTは聞いたことがある程度。

1. 現地の観光スポットでQRコードを見つける。「無料でデジタル記念品が受け取れます」という案内。
2. QRを読み込むとブラウザが開き、メールアドレスを入力する画面が表示される。
3. メールに届いた6桁のコードを入力するだけでログイン完了。
4. 位置情報の許可を求められ、承認すると「あなたはいま南城市◯◯スポットにいます」と確認され、NFTが受け取れる。
5. マイページに「南城市訪問記念」のデジタルバッジが表示される。旅行後も手元に残る。

---

### ストーリー②：Shopifyでチケットを購入してイベントに参加する

> **ユーザー：** 事前にネットで南城市のシュノーケリング体験を予約した旅行者。

1. Shopifyの予約サイトで体験チケットを購入する（通常のECと同じ操作）。
2. 購入後、メールアドレスに紐づいたウォレットへ自動でNFT（体験チケット）が発行される。
3. 当日、スタッフにスマートフォンを見せる。スタッフはNFTの「使用する」ボタンをタップして消込。
4. 使用済みになったNFTはグレー表示されるが、「◯月◯日に使用した」という記録として手元に残る。

---

### ストーリー③：スタンプラリー（クエスト）を楽しむ

> **ユーザー：** 何度も南城市を訪れているリピーター。コミュニティにも参加している。

1. マイページの「クエスト」タブを開くと、進行中のスタンプラリーが表示されている。
2. 各チェックポイントにあるQRコードをスキャンする。GPS位置情報が一致しないと無効になる仕組みで、現地に行かないとスタンプが押せない。
3. チェックポイントを通過するたびに、保有しているNFTの画像や名前が「進化」する演出が入る。
4. 全チェックポイントを制覇すると、限定のリワードNFTが贈られる。

---

### ストーリー④：観光客に地元住民がNFTをギフトする

> **ユーザーA：** 南城市在住の住民。観光客と話して仲良くなった。
> **ユーザーB：** 初めて南城市を訪れた旅行者。

1. 住民Aが自分のNFT詳細ページを開き、「譲渡する」ボタンをタップ。
2. QRコードが生成される。旅行者BのスマートフォンでそのQRを読み込む。
3. 旅行者Bのマイページに、住民AからのNFTが届く。
4. (推測) このような「人と人とのギフト体験」によって、旅行の思い出が具体的なデジタルオブジェクトとして残る設計。

---

### ストーリー⑤：有効期限が切れた住民証を「更新」して継続参加する

> **ユーザー：** 去年南城市のイベントで「デジタル住民証（SBT）」を受け取ったファン。

1. マイページを開くと、住民証がグレーになっていて「期限切れ」と表示されている。
2. 「更新する」ボタンを押すと、Shopifyの商品ページへ遷移する。
3. 更新料を支払うと、住民証の有効期限が自動でリセットされ、Discordのメンバーロールも継続される。
4. (推測) この仕組みにより「継続して関わるコスト」を意識させながら、継続の動機（コミュニティ参加、特典）を提供している。

## 3. ユーザー体験フロー（ざっくり）

### 初回利用時のフロー

```
【発見】
  QRコードを発見 / LINE友だち登録 / Shopifyで商品購入
      ↓
【認証】
  ログインページ
  ├─ メールアドレスを入力 → 6桁コードで本人確認（OTP）
  ├─ LINEブラウザから開いた場合 → 自動でLINE認証
  └─ 上級者向け：MetaMask等の外部ウォレット接続
      ↓
【受け取り】
  NFT受け取り画面（スポット / ドロップ / 購入自動発行）
  ├─ 位置情報確認が必要な場合 → GPS許可 → 距離チェック → 受け取り
  └─ 位置情報不要な場合 → 即時受け取り
      ↓
【確認】
  マイページ
  └─ 受け取ったNFTが表示されている
```

### 日常利用のフロー

```
マイページ（ダッシュボード）
  ├─ 保有NFTを確認 → NFT一覧 → NFT詳細
  │    ├─ チケットの消込（使用する）
  │    └─ 譲渡リンクの発行
  │
  ├─ クエスト（スタンプラリー）の進行確認
  │    └─ QRスキャン → GPS確認 → スタンプ獲得
  │
  ├─ コミュニティ連携確認
  │    ├─ Discord連携 → ロール自動付与
  │    └─ LINE連携確認
  │
  └─ アンケート回答 → リワードNFT取得
```

### 例外的なフロー（エラーや特殊ケース）

```
【有効期限切れ】
  期限切れNFT → グレー表示 → 「更新する」ボタン → Shopify再購入 → 期限リセット

【位置情報エラー】
  位置情報が取得できない / 距離が遠い → エラーメッセージ → ガイドページへ誘導
  （iPhoneとAndroidの設定方法を別ページで案内）

【ログアウト後】
  明示的にログアウトした場合、次回アクセス時にLINE自動ログインは発動しない
  （ユーザーが意図せず再ログインされることを防ぐ配慮）

【譲渡リンクのタイムアウト】
  譲渡リンクには有効期限があり、期限切れの場合は受け取り不可

【既にクレーム済みのリンク】
  同じ譲渡リンクを2回使おうとした場合はエラー表示
```

| **マイページ（ダッシュボード）** | `/mypage` | 自分の「デジタル住民としての現在地」を確認する場所 | 住民証・アセット数・コミュニティ連携状況の確認 |
| **NFT一覧** | `/mypage/nfts` | 自分が持っているデジタルアイテムを見渡す | カテゴリフィルタ（証明書・体験・資産・アート等）、使用済み/期限切れの識別 |

| **テンプレート配布** | `/drop/[templateId]` | イベントやキャンペーンのNFTを受け取る | NFT受け取り（位置情報不要の場合あり） |

| **通知センター** | `/mypage/notifications` | お知らせやアクティビティ履歴を確認する | 通知の確認 |

## 5. デザイン/UX上の意図が読み取れるポイント

### 入口の摩擦をとことん減らす

- **メールアドレスだけで完結する認証**：暗号資産ウォレットの作成・管理が不要。6桁のコードを入力するだけで使い始められる。
- **LINEブラウザからの自動認証**：LINE経由で訪問したユーザーには、改めてログイン操作を求めない（ただし、意図的にログアウトしたユーザーには発動しない配慮も入っている）。
- **位置情報が取れなかったときの専用ガイドページ**：エラーを出すだけでなく、「iPhoneの場合はここを設定してください」という手順案内ページへの誘導がある。

### 「使えなくなる」をちゃんと見せる

- **期限切れNFTをグレーにする**：有効期限が切れたNFTは視覚的に「使えない」ことが一目でわかる。ただし消えるのではなく、記録として残る。
- **使用済みチケットにタイムスタンプを残す**：「◯月◯日に使用しました」という履歴として保持される。思い出の記録としての価値。
- **「更新する」ボタンを目立つ場所に配置**：期限切れを検知したら、再購入への導線をわかりやすく表示する。

### 「ここに行かないとできない」をつくる

- **GPS確認が必須のNFT受け取り**：現地に行かないと受け取れない仕組みで、「現地体験の価値」を担保している。
- **スタンプラリーのリアル連動**：クエストのQRスキャンも位置情報で検証される。自宅から完了させることは（仕組み上）できない。

### コミュニティ所属感の可視化

- **Discordロールの自動付与・剥奪**：NFTを持っている間だけ、Discordのコミュニティで「メンバー」として認められる。期限が切れると自動的にロールが外れる。これにより「メンバーであり続けること」への動機が生まれる。
- **SBT（住民証）の所持者数を表示**：マイページに「現在◯人が持っています」という表示がある。(推測) コミュニティの規模感・自分がその一員であることを視覚的に伝える意図と考えられる。

### 「段階的に深まる関係」の設計

- **初回は無料・簡単に受け取れる**特典NFTで最初の体験を提供
- **継続参加には「更新費用」と「コミュニティへの貢献」が必要**という段階的な設計
- (推測) ランク機能（プラチナ → ダイヤモンド）が「開発中」として存在しており、将来的にはさらなる継続インセンティブの階層化が想定されている

### ギフト体験による関係性の広がり

- **譲渡リンク**（QRコード方式）で、ユーザーが自分のNFTを他の人に渡せる。これは単なる機能ではなく、「旅行者と地元住民が交流してギフトを渡し合う」という体験を設計の意図として含んでいると読める。

## 6. 不明点・コードだけでは判断できない部分

### ビジネスルール・運用方針について

- **「デジタル住民証」の価格設定と更新料**：有効期限の設定（validity_days）は管理者が決められるが、実際の価格・商品の組み合わせはShopifyの設定によるため、コードからは不明。
- **1テンプレートあたりの配布上限**：上限チェックのロジックは存在するが、具体的な数値はスタッフが設定するため、現行の運用値は不明。
- **無料で配布するNFTと有料のNFTの区別ルール**：どのNFTが無料でどれが有料になるかの判断基準はコードからは読み取れない。
- **アンケートへのリワードNFT付与ルール**：「アンケートを回答したらNFTを受け取れる」機能はあるが、すべてのアンケートにリワードがつくのか、一部だけなのかは設定次第。

### ユーザーの権限・ロールについて

- **ゲスト（未ログイン）の扱い**：ログイン前にどこまでサービスを見られるのか（例: 配布スポット一覧は見えるのか）は確認が必要。
- **LINEアカウントとメールアカウントの統合**：LINEでログインした人がメールでも登録した場合、同一ユーザーとして扱われるのか、別ユーザーになるのか。コード上は別々に見える部分があり、ユーザーが気づかず二重登録するリスクがある可能性がある。
- **スタッフ権限の粒度**：現状は「スタッフか否か」の二択。将来的に「テンプレート編集のみ」「配布のみ」などの権限分けが必要かどうかは要検討。

### 体験設計の意図確認が必要な点

- **NFTが「期限切れ」になったユーザーへのリマインダー通知**：通知機能は存在するが、「期限切れ直前にプッシュ通知を送る」ような自動リマインドがあるかどうかはコードからは確認できなかった。
- **「ランク」機能の仕様**：プラチナ・ダイヤモンドなどのランク表示コンポーネントは存在するが「開発中」と明示されており、現時点での確定仕様は不明。
- **クエスト完了後のユーザー動線**：クエスト完了でリワードNFTが付与された後、ユーザーに何を期待するのか（次のクエスト？SNSシェア？）の設計意図は確認が必要。
- **Discordに入っていないユーザーの体験**：NFTは持っているがDiscordを使わないユーザーに対して、コミュニティ参加を促す体験がどう設計されているかは不明。
- **多言語対応の実際の運用**：日本語と英語の両方に対応しているが、外国人観光客向けの導線（英語ページへの誘導）がどこでどのように機能するかは確認が必要。

## 付録：主要なNFTカテゴリと意味

| カテゴリ | 説明 | 特徴 |
|---|---|---|
| **証明書（SBT）** | 「デジタル住民証」。保有し続けることに意味がある | 有効期限あり、Discordロールと連動、譲渡不可 |
| **体験チケット** | イベント・ツアー等の参加証明 | 使用で消込（使用済みに変化）、Shopify購入で自動発行 |
| **資産** | デジタル価値を持つアイテム | (推測) 将来的な利用価値を想定した分類 |
| **アート** | 南城市ゆかりのデジタルアート作品 | 記念品としての性格が強い |
| **その他** | 上記に分類されないNFT | キャンペーン特典・限定品など |
